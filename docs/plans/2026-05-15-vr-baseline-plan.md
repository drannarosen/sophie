## Phase 2 — Extend test-runner config with screenshot logic (1 task, ~30 min)

### Task 2.1: Add `postRender` snapshot comparison to `test-runner.ts`

**Files:**
- Modify: `packages/components/.storybook/test-runner.ts`

**Step 1: Read the current file**

```bash
cat packages/components/.storybook/test-runner.ts
```

Understand: how axe-core is currently invoked in `postRender`. The screenshot logic appends to the same hook — does NOT replace axe.

The shape will be:

```ts
import type { TestRunnerConfig } from "@storybook/test-runner";
import { injectAxe, checkA11y } from "axe-playwright";   // existing
import { toMatchImageSnapshot } from "jest-image-snapshot";  // see Step 2

const customSnapshotsDir = `${process.cwd()}/__snapshots__/chromium`;

const config: TestRunnerConfig = {
  async preRender(page) {
    await injectAxe(page);                              // existing
  },
  async postRender(page, context) {
    await checkA11y(page, "#storybook-root", { /* existing axe options */ });

    // NEW: screenshot comparison.
    // Story title format: "ComponentName/StoryName" — derive the
    // file slug. Storybook's CSF makes context.id a string like
    // "components-aside--note" — use it directly.
    const image = await page.screenshot({ animations: "disabled" });
    expect(image).toMatchImageSnapshot({
      customSnapshotsDir,
      customSnapshotIdentifier: context.id,
      // Tighter threshold than ADR 0028's deferral run because
      // we're now Linux-vs-Linux (CI baseline vs CI run). Set 0
      // to start; loosen only if real CI flakes surface.
      failureThreshold: 0,
      failureThresholdType: "pixel",
    });
  },
};

export default config;
```

`jest-image-snapshot` provides `toMatchImageSnapshot` matcher. It needs to be registered.

**Step 2: Install `jest-image-snapshot` + types**

```bash
cd packages/components
pnpm add -D jest-image-snapshot@^6 @types/jest-image-snapshot@^6
```

**Step 3: Register the matcher with Vitest's `expect`**

`@storybook/test-runner` uses Jest-compatible expect. Add a setup file that extends expect:

Create `packages/components/.storybook/test-runner-setup.ts`:

```ts
import { expect } from "@storybook/test";
import { toMatchImageSnapshot } from "jest-image-snapshot";

expect.extend({ toMatchImageSnapshot });
```

Reference this from `test-runner.ts` by importing the setup at the top:

```ts
import "./test-runner-setup.ts";
```

**Step 4: Add TypeScript module declaration for the matcher**

Create `packages/components/.storybook/test-runner.d.ts`:

```ts
import "jest-image-snapshot";

declare module "vitest" {
  interface Assertion<T> {
    toMatchImageSnapshot(options?: import("jest-image-snapshot").MatchImageSnapshotOptions): void;
  }
}
```

(Adjust the module name if `@storybook/test-runner` uses a different expect provider — check its README. If it uses Jest's `expect`, declare `module "expect"` instead.)

**Step 5: Verify TypeScript compiles**

```bash
pnpm --filter @sophie/components typecheck
```

Expected: PASS. If TS errors on the matcher type, the `test-runner.d.ts` declaration is wrong — adjust the module name to match `@storybook/test-runner`'s expect provider.

**Step 6: Verify biome clean**

### Task 3.1: Run test-runner with `--updateSnapshot` to create initial baselines

**Files:**

```

**Step 2: Serve Storybook + run test-runner with --updateSnapshot**

In one terminal (or background):

```bash

```

Wait ~2s for the server to bind. Then:

```bash
cd packages/components

```

This iterates every story and generates a baseline PNG at `__snapshots__/chromium/<story-id>.png`. ~1-2 minutes for ~64 stories.

When done, kill the http-server.

**Step 3: Verify baseline files**

```bash
ls -1 packages/components/__snapshots__/chromium/ | wc -l
```

Expected: matches the story count (~64). If far less, something silently failed — check the test-runner output above for skipped stories.

```bash
du -sh packages/components/__snapshots__/chromium/
```

Expected: ~2-3MB total. If much larger, investigate (some story may have giant viewport or full-page screenshot).

**Step 4: Sanity-check a PNG**

Open one PNG via Read tool to verify it renders the expected component. Spot-check 3-4 PNGs across different component categories (Aside, KeyEquation, Search, LearningObjectives).

If any PNG is blank/black/missing content, debug:
- Storybook story may have a hydration race; add a `play` function that waits for the rendered state
- Or the snapshot was taken before fonts loaded; check `page.waitForLoadState("networkidle")` in pre-render

**Step 5: Add README.md**

Create `packages/components/__snapshots__/README.md`:

```markdown
# Component visual-regression snapshots

Baseline PNGs for [`@storybook/test-runner`](https://storybook.js.org/docs/writing-tests/test-runner). One PNG per story, captured in Chromium on CI's Ubuntu runner per [ADR 0057](../../../docs/website/decisions/0057-visual-regression-baseline.md).

## Regenerating baselines

Intentional visual changes require regenerating snapshots. Two paths:

**Recommended — CI regen:**
1. Push your branch with the component changes.
2. Once CI's `vr` job fails (diffs surface as artifacts), trigger:
   ```bash
   gh workflow run vr-update --ref <branch-name>
   ```
3. CI regenerates baselines and commits them back to the branch.
4. PR auto-updates; review the new PNGs in the Files changed tab.

**Local-only (will produce Mac-rendered PNGs that diverge from CI):**
- Build Storybook: `pnpm --filter @sophie/components build-storybook`
- Serve it: `pnpm dlx http-server packages/components/storybook-static --port 6006`
- Regen: `pnpm --filter @sophie/components test:vr:update -- --url http://127.0.0.1:6006`

Mac-local baselines fail in CI because of CoreText (macOS) vs FreeType (Linux) font rendering differences. Use the CI path for actual baseline updates.

## Structure

- `chromium/` — current Chromium baselines.
- (`firefox/`, `webkit/` reserved for future cross-browser coverage; not used in v1.)
```

## Phase 4 — CI workflow + regen workflow (1 task, ~20 min)

### Task 4.1: Add `vr` job to ci.yml + create vr-update.yml

**Files:**
- Modify: `.github/workflows/ci.yml`
- Create: `.github/workflows/vr-update.yml`

**Step 1: Read existing ci.yml**

```bash
cat .github/workflows/ci.yml
```

Find the `storybook` job to understand how it builds and what artifact it produces. The `vr` job depends on the same `storybook-static/` output.

**Step 2: Add `vr` job to ci.yml**

Append after the `storybook` job (or wherever new jobs go per the file's pattern). Skeleton:

```yaml
  vr:
    name: Visual regression (Chromium)
    needs: storybook
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - uses: actions/cache@v4
        with:
          path: ~/.cache/ms-playwright
          key: ${{ runner.os }}-playwright-${{ hashFiles('pnpm-lock.yaml') }}
      - run: pnpm exec playwright install chromium --with-deps
      - run: pnpm --filter @sophie/components build-storybook
      - name: Serve Storybook + run test-runner
        run: |
          pnpm dlx http-server packages/components/storybook-static --port 6006 --silent &
          sleep 5
          pnpm --filter @sophie/components test:vr -- --url http://127.0.0.1:6006
      - name: Upload diff artifacts on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: vr-diffs
          path: packages/components/__image_snapshots_diff__/
          retention-days: 7
```

Adjust details (caching, Node version, action versions) to match the existing jobs' patterns. **Match the existing jobs' style verbatim** — don't introduce new caching or setup patterns this PR doesn't need.

The `__image_snapshots_diff__` path is `jest-image-snapshot`'s default output dir when a snapshot fails. Verify by inducing a failure locally (touch a component's CSS, re-run `test:vr`) and seeing where the diff PNG lands.

**Step 3: Create `.github/workflows/vr-update.yml`**

```yaml
name: VR Update
on:
  workflow_dispatch:
    inputs:
      branch:
        description: Branch to regenerate baselines on
        required: true
        type: string

jobs:
  update:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ inputs.branch }}
          token: ${{ secrets.GITHUB_TOKEN }}
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec playwright install chromium --with-deps
      - run: pnpm --filter @sophie/components build-storybook
      - name: Regenerate baselines
        run: |
          pnpm dlx http-server packages/components/storybook-static --port 6006 --silent &
          sleep 5
          pnpm --filter @sophie/components test:vr:update -- --url http://127.0.0.1:6006
      - name: Commit regenerated baselines
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add packages/components/__snapshots__
          if git diff --staged --quiet; then
            echo "No baseline changes — VR is already in sync."
          else
            git commit -m "chore(vr): regenerate baselines via vr-update workflow"
            git push origin HEAD:${{ inputs.branch }}
          fi
```

**Step 4: Validate YAML syntax**

```bash
pnpm dlx yaml-lint .github/workflows/ci.yml .github/workflows/vr-update.yml 2>&1
```

Or use any quick YAML parser. If invalid YAML, fix before commit.

**Step 5: Commit**

```bash
git add .github/workflows/ci.yml .github/workflows/vr-update.yml
git commit -m "ci(vr): add vr job + vr-update workflow per ADR 0057"

**Step 6: Push branch + verify CI runs the vr job**

If the `vr` job fails on first push despite baselines existing, the most likely cause is a tooling-version mismatch between local generation and CI. Investigate before proceeding.
