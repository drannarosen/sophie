# Visual regression reference

The API surface for Sophie's visual-regression (VR) infrastructure. The architectural decision lives in [ADR 0057](../decisions/0057-visual-regression-baseline.md); the dev-loop recipes live in the [Run visual regression locally](../how-to/run-visual-regression-locally.md) how-to. This page is the lookup table.

## Snapshot directory layout

```text
packages/components/__snapshots__/
├── README.md                                       # operational notes adjacent to the files
└── chromium/                                       # Chromium baselines (current canonical browser)
    ├── components-aside--note.png                  # one PNG per story
    ├── components-keyequation--with-block-math.png
    ├── components-search-searchmodal--default.png
    └── ...                                         # 68 PNGs total as of 2026-05-16
```

- **Filename convention**: `<storybook-csf-id>.png` where the CSF id is `<group-slug>--<story-slug>` (Storybook derives this deterministically from the story's `title` + export name).
- **Browser subdirectory**: `chromium/` is the only one populated; `firefox/` and `webkit/` are reserved for future cross-browser coverage. Not used in v1.
- **Generated by**: the Linux CI runner via the `vr-update` workflow. Mac-local regen is structurally impossible (CoreText vs FreeType rendering divergence).
- **Tracked**: as regular git binaries, ~30–60 KB each, ~1.4 MB total. No git-lfs.

## Test-runner config

File: [`packages/components/.storybook/test-runner.ts`](https://github.com/drannarosen/sophie/blob/main/packages/components/.storybook/test-runner.ts)

Three lifecycle hooks compose three responsibilities:

| Hook | Runs in | Purpose |
| --- | --- | --- |
| `setup()` | Jest worker | Registers the `toMatchImageSnapshot` matcher via `expect.extend()`. Must be in `setup()`, not at module top-level, because the test-runner CLI loads `test-runner.ts` in a Node process where `expect` is not a global. |
| `preVisit(page)` | Jest worker (per story) | Injects axe-core into the page via `axe-playwright`'s `injectAxe()`. |
| `postVisit(page, context)` | Jest worker (per story) | Runs axe (`checkA11y`) against `#storybook-root`, then waits for fonts/images via `waitForPageReady`, then takes a Chromium screenshot and compares to baseline via `toMatchImageSnapshot`. |

Both axe and VR run on every story unless opted out. Failures in either block CI.

## Environment variables

| Var | Values | Effect |
| --- | --- | --- |
| `SKIP_VR` | `1` (set) / unset | When set, the `toMatchImageSnapshot` block in `postVisit` is skipped. Axe still runs. Used by the `test:storybook` package script to enable Mac-local axe validation; CI's `storybook` job inherits the same behavior. |

`SKIP_VR` is the recommended way to run axe-only locally. Don't set it in the `vr` CI job — that job's job is to run both gates.

## Story parameters

| Parameter | Effect | When to use |
| --- | --- | --- |
| `parameters.a11y.disable: true` | Skips the axe block for this story. | Genuine a11y exemptions (e.g., a story rendering an intentionally-bare element for visual test). Document the reason. |
| `parameters.vr.disable: true` | Skips the VR block for this story. Axe still runs. | Non-deterministic stories: live timestamps, randomized fixtures, animation mid-frames. Document the reason. |

Example:

```tsx
export const NonDeterministic: Story = {
  parameters: {
    vr: { disable: true },   // story renders Date.now(); diffs aren't meaningful
  },
  args: { /* ... */ },
};
```

## CI jobs

Both defined in [`.github/workflows/ci.yml`](https://github.com/drannarosen/sophie/blob/main/.github/workflows/ci.yml).

### `storybook` job

- Builds `storybook-static/` via `pnpm exec turbo run build-storybook --filter=@sophie/components`.
- Serves it on port 6006 via `http-server`.
- Runs `pnpm --filter @sophie/components test:storybook`, which has `SKIP_VR=1` baked into the script.
- Result: **axe-only gate.** Fails on a11y violations; doesn't see VR diffs.

### `vr` job

- Same setup as `storybook`.
- Runs `pnpm --filter @sophie/components test:vr` (no `SKIP_VR`).
- Result: **axe + VR gate.** Fails on a11y violations OR pixel diffs.
- On failure: uploads `vr-diffs` artifact containing `__snapshots__/chromium/__diff_output__/<story-id>-{expected,actual,diff}.png`.

### Why two jobs?

The shared `test-runner.ts` composes axe + VR in one `postVisit` hook. Before [PR #58](https://github.com/drannarosen/sophie/pull/58), the `storybook` job ran both, so any visual diff failed it too — two red checks reporting the same bug. PR #58 introduced `SKIP_VR=1` and split the responsibilities: `storybook` gates a11y, `vr` gates a11y + visuals.

The history is preserved here so future contributors don't re-couple them in a "simplification" refactor.

## `vr-update` workflow

File: [`.github/workflows/vr-update.yml`](https://github.com/drannarosen/sophie/blob/main/.github/workflows/vr-update.yml).

Manually triggered. Regenerates baselines on CI Linux and commits them back to the branch.

### Trigger

```bash
gh workflow run vr-update --ref <branch-name> --field branch=<branch-name>
```

Both arguments are load-bearing:

- `--ref <branch-name>` — tells GitHub Actions which checkout to run the workflow from.
- `--field branch=<branch-name>` — passed to the workflow as input, telling it which branch to push regenerated PNGs back to.

The bare `--ref` form (without `--field branch=`) fails silently — the workflow runs but skips the push step because `inputs.branch` is empty.

### What it does

1. Checks out the input branch.
2. Builds Storybook (`pnpm exec turbo run build-storybook`).
3. Installs Playwright Chromium (`pnpm exec playwright install --with-deps chromium`).
4. Serves Storybook on 127.0.0.1:6006.
5. Runs `pnpm --filter @sophie/components test:vr:update`, which regenerates every story's baseline PNG.
6. Commits `packages/components/__snapshots__` as `github-actions[bot]` with message `chore(vr): regenerate baselines via vr-update workflow`.
7. Pushes back to the input branch via `${{ secrets.GITHUB_TOKEN }}`.

Typical runtime: 2 minutes.

### The CI re-run gotcha

The bot push uses `GITHUB_TOKEN`. By design, pushes from `GITHUB_TOKEN` do not fire `pull_request: synchronize` events on PRs targeting that branch — GitHub's loop-prevention mechanism. The `vr` check on the PR won't auto-rerun.

To trigger a fresh CI run on the bot commit, **close and reopen the PR**:

```bash
gh pr close <pr-num>
gh pr reopen <pr-num>
```

This fires `reopened`, CI runs against the new HEAD (the bot commit), and the `vr` check goes green.

## Known limitations

### Mac-local `pnpm test:vr` always fails

Structural per [ADR 0057](../decisions/0057-visual-regression-baseline.md). Mac uses CoreText for font rasterization; Linux (CI) uses FreeType. The same HTML/CSS renders to byte-different PNGs even at identical viewports. Don't try to fix this by regenerating baselines locally — your Mac PNGs will then fail in CI. The recommended local loop is `test:storybook` (axe only) plus push-to-CI for VR validation.

The Docker-on-Mac escape hatch (run an Ubuntu container that matches CI exactly) is named in ADR 0057's consequences section as the natural escalation if push-to-CI feels too slow. Not implemented yet; defer until friction warrants.

### Popover-default-closed stories don't capture popover content

Tracked as [issue #57](https://github.com/drannarosen/sophie/issues/57). Components like `<EquationRef>` render an inline pill that opens a popover containing the actual content (KaTeX math, in EquationRef's case). The popover defaults to closed, so the captured baseline shows only the trigger pill. The popover-open visual contract is uncovered.

Fix path: add a Storybook `play()` function to the story that clicks the trigger before the test-runner snapshots. Out of scope for the VR-baseline PR (#53); tracked for Workstream 3 component-polish PRs.

### One-pixel diffs in font-asset-dependent stories

When Storybook's bundle gains or loses a font file (e.g., the KaTeX font assets joined the bundle in [PR #53's KaTeX-rendering fix](https://github.com/drannarosen/sophie/pull/53)), unrelated stories may shift by sub-pixel amounts due to font-loading order changes. The baselines naturally update on the next `vr-update` run; no action needed unless a specific story shifts visibly.
